# planner-shell - Practical AI DevOps Agent

> **⚠️ IMPORTANT WARNING**: planner-shell is designed **exclusively** for creating disposable test environments. It is **NOT intended for production use**. The agent executes AI-generated commands which may **destroy your environment**. Use **entirely at your own risk**.

`planner-shell "install Apache Superset and load sample data"` — With just this command, the AI ​​agent searches the web for information relevant to its OS type, devises an execution plan, and carries out everything from installation to loading sample data.

_**"A DevOps agent that doesn't fear failure, learns, and accumulates the team's wisdom."**_

**Safety** that can be verified by a human before command execution (human-in-the-loop), **Efficiency** that avoids repeating the same search task on the web, **Autonomy** that tries to fix failures itself, and **Collective Intelligence** that stores continuously evolving deployment plans in a shared database called **SingleStore**  (optional).


## Quick Start

```bash
git clone https://github.com/bobo3977/planner-shell.git && cd planner-shell && chmod +x setup.sh && ./setup.sh
```

Edit `.env`: (at least OPENAI_API_KEY or OPENROUTER_API_KEY is required)

```env

# Required: One of these (OpenAI or OpenRouter)
```

Run:

## Core Concept: Planning × Auditing × Memory × Execution

**planner-shell** is composed of four key elements:

1. **Planner (Planning Agent)**:
   Uses Tavily search to gather up-to-date information and generates execution plans via LLM.

2. **Auditor (Command scan module)**:
   Scans the plan generated by Planner for dangerous commands (e.g., `rm -rf`) and issues warnings if found.

3. **Cache (Memory)**:
   Stores generated plans and enables reuse for future identical or similar tasks through **hybrid search** combining vector search and BM25 full-text search. This saves both LLM tokens and Tavily tokens.

4. **Executor (Execution Agent)**:
   Executes plans one by one in a persistent PTY shell, observes results, and autonomously fixes errors when they occur.

## Key Features

- **Persistent PTY Shell**: Single bash session with full state persistence (`cd`, `export`, environment variables)
- **Human-in-the-Loop**: Approve, edit, skip, or quit each command before execution
- **Smart Caching**: Pluggable deployment plan cache backend (SQLite or SingleStore) with configurable TTL
- **Hybrid Search**: Combines BM25 (FTS5) and vector search (embeddings) for cached deployment plan retrieval

## Usage

### Interactive Mode

Run the agent without arguments:

```bash
planner-shell
```

Enter a task description, URL, or markdown file path:

```
🤖 Enter your task (or 'exit'): Install Apache Superset and load sample data
```

#### Quick Reference

```
📋 Quick Guide:
  • Enter a task description (e.g., 'Install Nginx')
  • Enter a URL to fetch content (e.g., 'https://example.com/guide')
  • Enter a markdown file path (e.g., './setup.md')
  • 'ls' — list cached plans
  • 'run N' or 'rN' — run cached plan by index (e.g., 'run 1', 'r1')
  • 'run <text>' or 'r <text>' — run cached plan by task match (e.g., 'run nginx')
  • 'refine' — distil last execution into a clean plan
  • Type 'quit' or 'q' to exit
```

The agent operates in two phases:

1. **Planning Phase**: Creates a detailed execution plan
   - Uses Tavily search if API key is available (unless URL/Markdown mode)
   - Without Tavily: uses LLM's internal knowledge only (degraded mode)
   - Caches plans for future reuse (30-day TTL)
   - Shows similar cached plans if available (hybrid search, except for URL/markdown inputs)

2. **Execution Phase**: Runs commands step-by-step
   - Each command requires approval before execution
   - Real-time output streaming
   - AI analyzes results and proceeds accordingly
   - Automatic error handling using Tavily search

### Approval Options

When a command is proposed, you can:

- `Y`/`Enter` - Execute as-is
- `N` - Deny and provide feedback to the AI
- `edit` or `e` - Open in editor (vim/nano) to modify the command
- `skip` or `s` - Skip (treated as success, no execution)
- `finish` or `f` - End plan execution early (remaining steps marked as done)
- `quit` or `q` - Exit the agent immediately and return to shell

### History Navigation (back/skip)

During command approval, you can navigate through previously executed commands:

- `back` or `b` - Go back to a previous command in history
  - Shows the command with its exit code and output
  - You can then execute, edit, skip, or go back further
  - Skipped commands are also shown (marked as skipped)
- `skip` or `s` (when viewing history) - Mark the current historical command as skipped and move forward
  - The skipped command remains in history with a "skipped" flag
  - After skipping, you'll see the next (newer) non-skipped command
  - If all newer commands are skipped, you'll return to the AI-proposed command


### Plan Editing

After the plan is generated, you can edit it before execution:

```
✅ Plan generated. Execute? [Y/n/edit]:
```

- Press `e` or `edit` to open the plan in your editor (vim/nano)
- After editing, you'll be asked: `💾 Save edited plan to cache? [Y/n]:`
  - `Y`/`yes` - Save the edited plan to cache for future reuse
  - `n`/`no` - Keep the original cached plan (if any), use edited plan for this execution only
- The edited plan is displayed for review before execution

This allows you to:
- Fine-tune AI-generated plans before execution
- Add, remove, or modify steps
- Save improved plans to cache for future tasks

### Special Commands

During the task prompt, these commands are recognized:

- `exit` / `quit` / `q` - Exit the agent
- `clear` / `cc` - Clear all cached plans
- `cleanup` / `expired` - Remove expired cache entries (older than 30 days)
- `refine` - Distill a clean plan from the execution log
- `list` / `ls` - List cached plans (supports shortcuts like `e1`, `d5` in the pager)
- `edit <idx>` / `e<idx>` - Edit a cached plan by index
- `delete <idx>` / `d<idx>` - Delete a cached plan by index
- `run <idx>` / `r<idx>` - Run a cached plan directly by index (e.g., `run 1`, `r1`)
- `run <task>` / `r <task>` - Run a cached plan by task text match (e.g., `run nginx`, `r nginx`)

### URL/Markdown Mode

Execute documented procedures from external sources:

```
🤖 Enter your task: https://example.com/install-guide
# or
🤖 Enter your task: ./setup.md
```

The agent:
- Fetches the URL content or reads the markdown file
- Creates a plan using ONLY the provided content (no Tavily search)
- Executes the documented procedure step-by-step
- Useful for running installation guides, tutorials, or SOPs

### Auto-Approval Mode (Non-Interactive)

For automation, CI/CD, or unattended execution:

```bash
# Direct URL or markdown file
AUTO_APPROVE=1 planner-shell https://example.com/guide
AUTO_APPROVE=1 planner-shell ./setup.md

# Interactive task with auto-approval
AUTO_APPROVE=1 planner-shell
```

In this mode:
- All commands are auto-approved (simulates pressing 'Y')
- No interactive prompts
- Ideal for automated testing or scheduled tasks
- Still respects safety features (non-interactive flags)

### Ctrl+C Handling

- **Single Ctrl+C**: Interrupts the current command (sends SIGINT to the shell), plan continues
- **Double Ctrl+C** (within 2 seconds): Aborts the entire execution and returns to prompt

## Plan Distillation

After successful execution, you can extract a clean plan from the execution log:

```bash
Type 'refine' to distill a clean plan from the execution log
```

This feature:
- Analyzes successful execution paths
- Removes errors and failed attempts
- Creates a simplified, error-free plan
- Stores the distilled plan in cache for future reuse
- Shows visual differences between original and distilled plans

## Architecture Deep Dive

### Persistent PTY Shell

The agent maintains a single bash process via a pseudo-terminal (PTY) using Python's `pty` module. This ensures:

- **State Persistence**: `cd` changes directory, `export` sets environment variables, and shell variables persist across commands
- **Interactive Command Support**: Commands that require a TTY (like `sudo`, `ssh`) work correctly
- **Signal Handling**: Ctrl+C (SIGINT) is properly forwarded to the child process
- **Real-time Output**: Command output is streamed to the terminal as it's produced

The shell is implemented in the [`PersistentShell`](shell/persistent.py) class with automatic restart on crashes.

#### Known Limitations

- **`newgrp` Command**: The `newgrp` command starts a new shell session which breaks PTY control and can cause terminal display issues. When you need to run commands with a different group:
  - Use `sg <group> -c "<command>"` to execute a single command in a new group
  - Use `newgrp <group> && <command>` to switch groups and execute immediately
  - The most reliable method is to log out and back in with the new group

### Three-Agent System

**Planner Agent** ([`PlannerAgent`](agents/planner.py)):
- Creates detailed, numbered execution plans
- Uses Tavily search for up-to-date information if API key is available (unless URL/Markdown mode)
- Without Tavily: relies on LLM's internal knowledge (degraded mode)
- Implements hybrid cache lookup: exact hash match → FTS5 + vector similarity → generate new
- For URL/markdown inputs: skips hybrid search, uses exact hash match only
- System prompt enforces: specific commands, non-interactive flags, OS-aware instructions

**Executor Agent** ([`ExecutorAgent`](agents/executor.py)):
- Receives a pre-built plan
- Executes commands one-by-one using [`PersistentShellTool`](shell/tool.py)
- Analyzes output after each command
- Handles failures by searching remediation using Tavily
- Enforces max iterations to prevent infinite loops

**Auditor** ([`Auditor`](agents/auditor.py)):
- Scans proposed execution plans for potentially dangerous commands (e.g., file system destruction, privilege escalation)
- Compares commands against a custom list of patterns (configurable via `config/dangerous_patterns.txt`)
- Provides clear warnings and line numbers for detected risks
- Enabled by default; can be toggled via `ENABLE_AUDITOR=1/0`

### Plan Cache System

The agent supports pluggable cache backends via the [`BasePlanCache`](cache/base.py) abstract class:

#### SQLite Backend (Default)

The [`SQLitePlanCache`](cache/sqlite.py) uses SQLite with three lookup strategies:

1. **Exact Match** (O(1)): SHA-256 hash of normalized task text
2. **Hybrid Search**: Combines (skipped for URL/markdown inputs):
   - **FTS5 BM25**: Full-text search on task descriptions
   - **Vector Similarity**: Cosine similarity using OpenAI embeddings (`text-embedding-3-small`)
   - Weighted blend (α=0.6 favors vectors, configurable via `ALPHA`)
3. **TTL**: 30-day expiration (configurable via `ttl_days`)

The cache automatically:
- Detects database corruption and rebuilds with backup
- Maintains an FTS5 virtual table with triggers for sync
- Skips embedding generation for URL/markdown-based tasks (via `skip_embedding`)

#### SingleStore Backend (Optional)

The [`SingleStorePlanCache`](cache/singlestore.py) provides the same functionality with a cloud-native distributed SQL database:

- **Storage**: Table with `task_hash`, `task_text`, `plan`, `timestamp`, `embedding` (JSON)
- **Full-Text Search**: Uses SingleStore BM25 function for text matching (via FULLTEXT index)
- **Vector Search**: Dot Product on embedding vectors stored as Vector type
- **Hybrid Scoring**: RRF (Reciprocal Rank Fusion)
- **TTL**: Configurable expiration (default 30 days)
- **High Availability**: Benefits from SingleStore's distributed architecture

**Configuration**: Set `PLAN_CACHE_BACKEND=singlestore` and provide connection details via environment variables (see [Configuration](#configuration) section).

**Note**: SingleStore backend requires the `singlestoredb` Python package and a running SingleStore database.

### Output Management

- **Summarization**: Long command output (>20 lines) is truncated to head+tail (10 lines each) to limit LLM tokens
- **Streaming**: Output is streamed to terminal in real-time, with markers hidden
- **Exit Code Remapping**: Exit codes 130/131 (Ctrl+C) are remapped to 0 so the agent continues

### Safety & Robustness

- **Non-Interactive Flags**: Automatic injection of `--no-pager` for systemd commands, `-y` for apt/dnf/pacman
- **DPkg Lock Timeout**: Adds `-o DPkg::Lock::Timeout=60` to apt commands to handle concurrent installs
- **Idempotency Encouragement**: System prompt advises checking existing state before creating/starting
- **Graceful Degradation**: Falls back from `langchain_tavily` to `langchain_community` imports

```env
PLAN_CACHE_SCORE_THRESHOLD=0.0  # Minimum score threshold

# Cache TTL (days)
PLAN_CACHE_TTL_DAYS=30
```

#### SingleStore Settings (when PLAN_CACHE_BACKEND=singlestore)

## Performance Tips

1. **Cache Warm-up**: Repeated tasks (e.g., "install Python 3.11") will hit the cache after first run (instant)
2. **Hybrid Search**: When cache miss occurs, similar plans are offered to reduce API costs (not used for URL/markdown)
3. **Embedding API**: Vector search requires OpenAI API calls; if unavailable, falls back to pure FTS5
4. **Thread Timeout**: Agent threads have configurable timeouts (default: 30 minutes for executor)
5. **Periodic Cleanup**: Expired entries are cleaned hourly; run `cleanup` manually to reclaim space immediately
